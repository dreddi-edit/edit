External Dependencies & Service Map

Technische Gesamtuebersicht aller externen Dienste, Secrets und Betriebsabhaengigkeiten fuer das Projekt. Stand basiert auf aktueller Code- und Env-Analyse.

Runtime: Node.js 22.12.0 via Nixpacks
Platform: Railway (Port 8080, Healthcheck /health)
DB Engine: SQLite (better-sqlite3, WAL enabled)
Dokument: status/external_dependencies_visual_map.html

Storage & Persistence state

  • SQLite file path: DB_PATH (default server/data/editor.db)
  • WAL journaling enabled for concurrency
  • Optional cloud object store hooks: Google Cloud Storage (@google-cloud/storage)
  • Screenshot/render exports use Puppeteer and filesystem output
  • Thumbnails/assets in repository folders plus runtime write paths

Operational Flow (Kurzform)

1) Client

Dashboard ruft API auf und nutzt oeffentliche Frontend Keys (VITE_*) nur fuer browserseitige Integrationen.

2) Server API

Express Routes lesen Secrets aus ENV. OAuth Callback erzeugt Session Cookies/JWT. AI und Stripe laufen serverseitig.

3) Externe Provider

Google/GitHub/Facebook/X/Figma, Anthropic/Gemini/Groq, Stripe, Resend, optional GCP Services.

4) Persistenz

SQLite speichert User, Projekte, Billing, Audit Logs und ggf. User API Keys. Datenpfad ueber DB_PATH.

Browser -> Express API -> External APIs -> SQLite / Filesystem

Secret / Variable Wofuer genutzt Scope Storage Empfehlung Risiko
ANTHROPIC_API_KEY Claude Requests im Backend Server only Railway Secret Variable hoch
GEMINI_API_KEY / GROQ_API_KEY Alternative AI Provider Server only Railway Secret Variable hoch
STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET Zahlungen + Webhook Validierung Server only Railway Secret Variable hoch
GOOGLE_CLIENT_SECRET, GITHUB_CLIENT_SECRET, ... OAuth Login Callback Austausch Server only Railway Secret Variable hoch
JWT_SECRET, REFRESH_TOKEN_HASH_SECRET Token Signatur und Refresh Schutz Server only Railway Secret Variable hoch
DB_PATH Pfad zur SQLite Datei Server runtime Railway ENV + persistent volume mount path mittel
VITE_* Keys Frontend Integrationen (oeffentlich sichtbar) Client-visible Nur fuer public-safe APIs verwenden mittel
ALLOWED_ORIGIN CORS Einschrankung Server runtime Railway ENV niedrig

Railway Volume & DB Empfehlung

  • SQLite braucht persistentes Dateisystem, sonst Datenverlust beim Redeploy.
  • Empfehlung: Railway Volume mounten und DB_PATH auf den Mount zeigen (z. B. /data/editor.db).
  • Ohne Volume liegt server/data/editor.db im ephemeren Container-Dateisystem.
  • Backups/Export fuer kritische Daten einplanen.
